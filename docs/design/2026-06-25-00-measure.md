# Design: `sonda measure`

## Starting principle

Gall's Law: a complex system that works evolved from a simple system
that worked. We start with the simplest useful measurement primitive
and grow from there.

## Shape of a measurement command

- Each `sonda measure ...` subcommand is a standalone CLI tool
with flags controlling its behavior. No positional arguments.

- Each primitive should fail in exactly one way. For example,
`sonda measure dns over https` takes IP:port, SNI, HTTP host,
and URL path as separate flags rather than a domain name that
would need resolving first. The caller composes the resolution
and the DoH query as separate steps.

- Flags use `vflag` with `@DEFAULT_VALUE@` interpolation so that
`--help` always shows current defaults. Flag names are lowercase
kebab-case (e.g., `--query-type`, `--http-host`, `--url-path`).

## Output contract

- Primary output: structured NDJSON logs on stdout, one JSON
object per line. Exit code signals success (0) or failure (1)
or usage error (2).

- All log entries in a measurement share a `spanID` (UUIDv7)
for correlation.

- Log messages are single camelCase words (e.g., `measurementStart`,
`dialFailed`, `responseA`) to simplify downstream filtering.

- Every measurement emits `measurementStart` first and
`measurementDone` last (via defer). These bookends give stream
consumers confidence about where a measurement begins and ends.

- `commandLineArgs` is logged immediately after `measurementStart`,
including the full `sonda measure ...` prefix, so that the exact
command can be reproduced.

- Optional: raw protocol bodies (HTTP request/response) can be
saved separately for debugging (e.g., `-o/--output <file>`). Not
implemented yet: YAGNI for now.

## Processing is out of scope

- The consumer of structured logs and exit codes is a separate
component. `sonda measure` does not aggregate, retry, loop,
or decide what to do with results. It measures once and exits.

## Why CLI-shaped primitives

- The final system will probably not shell out to these commands.
But designing each measurement as a self-contained CLI allows
fast experimentation and evolution without coupling.

- Bonus: users can invoke measurements in ways we did not
anticipate, compose them with shell pipelines, or integrate
them into their own tooling.

## What we know about the goal

- The goal is to measure quality of experience (QoE) across
multiple layers: DNS, TLS, HTTP, streaming, throughput.

- Each primitive measurement layer becomes a `sonda measure ...`
subcommand following the same pattern established here: for now we
have DNS over UDP and DNS over HTTPS but we will be able to
extend this to other protocols relatively easily.
