# Design: logging policy

## Two categories of commands

Sonda commands fall into two groups with different logging needs:

- **Measurement commands** (`sonda measure *`) produce structured
  JSON logs to stdout. These logs ARE the output — consumers parse
  them as `stdout.txt` inside the spool. The schema is documented
  in the structured log discipline design doc.

- **Operational commands** (`sonda spool *`, `sonda scan`) produce
  human-facing diagnostic output to stderr. No consumer parses
  this output programmatically.

## Handler choice

Both groups use `log/slog`. The difference is the handler:

| Group       | Handler       | Output | Message style             |
|-------------|---------------|--------|---------------------------|
| measure     | `JSONHandler` | stdout | camelCase keys, no spaces |
| spool, scan | `TextHandler` | stderr | normal text sentences     |

Measurement commands use `JSONHandler` because the output is a data
contract: parsers match on `msg` values and extract typed fields. Keys
like `httpRoundTripDone` are identifiers, not prose.

Operational commands use `TextHandler` because the audience is a
human reading stderr. Messages like `"failed to create spool
directory"` are more useful than `"mkdirAllFailed"` when scanning
terminal output.

## Why slog for operational commands

Before this policy, the spool commands used `fmt.Fprintf` and scan
used `slog.JSONHandler`. Both had drawbacks:

- `fmt.Fprintf` has no level concept. Every message looks the same
  regardless of severity. Adding verbosity control later would
  require threading a custom flag through every function.

- `JSONHandler` on stderr is noisy for human consumption. The
  timestamp, level, and message are buried in JSON syntax.

`slog.TextHandler` gives leveled output (Error, Warn, Info, Debug)
in a readable format:

```
time=... level=INFO msg="extracted metrics" spanDir=... rows=4
time=... level=WARN msg="failed to parse span" spanDir=... err=...
```

This also means a future `--verbose` flag only needs to lower the
level from Info to Debug: the plumbing is already there.

## Handler options

Operational commands pass `nil` options to `NewTextHandler`, which
defaults to `LevelInfo`. This suppresses Debug-level messages
unless the handler is explicitly reconfigured. Measurement commands
set `LevelDebug` because all emitted events are meaningful data,
not diagnostic noise.

## What this is not

- Not a logging framework. There is no custom logger type, no
  wrapper, no middleware. Each command creates a `slog.Logger`
  with the appropriate handler and uses it directly.

- Not a change to the structured log schema. Measurement output
  format is unchanged. Only operational stderr logging changed.
