# Design: `internal/netstack`

## Purpose

High-level network operations (DNS resolution, IP address lookup)
built on top of sonda's CLI measurement primitives. Consumers use
Go types with familiar signatures; the package handles spool
plumbing and output parsing internally.

## Layering

- `SondaMeasurer` is the execution layer. It builds a
`sonda spool run -- sonda measure ...` command, executes it,
and returns the span directory path. It does not parse output.

- Above it, types like `Resolver` and `STUNLookupper` call
`SondaMeasurer.Run` with the appropriate operation, then parse
`stdout.txt` to extract results.

- `SondaOperation` is the interface between the two: each
operation type (`SondaMeasureDNSOverUDP`, `SondaMeasureSTUN`,
etc.) knows how to build its command-line arguments.

## API compatibility with the standard library

- `Resolver.LookupHost(ctx, domain) ([]string, error)` matches
the signature of `net.Resolver.LookupHost`. `LookupA` and
`LookupAAAA` are the building blocks underneath.

- This is deliberate: if consumer code depends on these
signatures, the underlying implementation can be swapped from
subprocess-via-spool to in-process without breaking callers.

## Observability as a side effect

- Every operation produces a spool entry with `argv.json`,
`stdout.txt`, `stderr.txt`, and `exitcode.txt`. Observability
is a consequence of the architecture, not an extra feature.

## Output parsing contract

- Parsing relies on the NDJSON output contract from `sonda
measure`: each line is a JSON object, the `msg` field identifies
the event type, and result data lives in known fields (`value`
for DNS, `ip`/`port` for STUN).

- Parsers scan line-by-line, skip entries they do not recognize,
and return an error when no matching result is found.

## Executable override

- `SondaMeasurer.Executable` can be set to an explicit binary
path. When empty, it falls back to `os.Executable()`. The
override exists because `os.Executable()` returns the calling
binary, which is only correct when called from `sonda` itself.

## Shared path knowledge

- Spool directory structure (`SpanDir`, `SpanStdout`, etc.)
lives in `internal/paths`, shared between `spool run` (which
writes) and `netstack` (which reads). Neither package imports
the other.

## What is not here yet

- No retry, concurrency, or scheduling. This package runs one
operation at a time and returns the result. Composition lives
above it.

- No DoH resolver variant. Adding one is mechanical: create a
new `Resolver`-like type that uses `SondaMeasureDNSOverHTTPS`
instead of `SondaMeasureDNSOverUDP`.
