# Design: `sonda spool gc`

## Purpose

Removes stale span directories from the spool. Without periodic
cleanup the spool grows without bound, since `sonda spool run`
creates a new directory for every measurement invocation.

## How it works

The command walks the spool sharding tree (`XXXX/X/X/<spanID>`)
and removes every span directory whose UUIDv7 timestamp is older
than a configurable cutoff (default: 6 hours). Both final and
`.tmp` (crash-orphaned) directories are eligible for removal.

After removing spans, the walk prunes empty sharding directories
on the way back up, so the tree stays compact.

## Flags

- `--spool-dir DIR` — root of the spool tree (default: `.`).
- `--max-age DURATION` — remove spans older than this (default: `6h`).

## Integration with `sonda scan`

`sonda scan` invokes `sonda spool gc` as a subprocess at the end
of each scan cycle, forwarding its own `--spool-dir` and
`--spool-max-age` values. This keeps the spool bounded without
requiring a separate cron job or timer. GC failure is logged as
a warning but does not fail the scan (unless `--fail` is set).

## Why a separate subcommand

- Keeps `spool run` and `scan` free of cleanup logic.
- Users and operators can run GC independently (different
  retention, ad-hoc cleanup, dry-run wrapper) without touching
  the measurement path.
- Testable in isolation: no dependency on the measurement stack.

## Safety properties

- Only directories whose name (minus `.tmp` suffix) parses as a
  valid UUIDv7 are considered. Other entries are silently skipped.
- `os.Remove` on sharding directories only succeeds when they
  are empty, so a concurrent `spool run` writing into the same
  shard cannot lose data.
- A span that is still being written lives under a `.tmp` name
  and will only be removed if its UUIDv7 timestamp is older than
  the cutoff — normally impossible unless the writing process has
  been stuck for longer than `--max-age`.

## What this is not

- Not a background daemon. GC runs once and exits.
- Not a data-retention policy. The default 6-hour window is
  sized for the current "collect and inspect" workflow. A real
  retention policy will live in whatever ingestion pipeline
  eventually consumes the spool.
