# Design: `sonda scan`

## Purpose

Periodic scan of specific network endpoints. Intended to be
triggered by a systemd timer (or equivalent). Runs a fixed
sequence of measurements and stores all results in the spool.

## Current shape

The scan performs these steps in order:

1. Resolve `stun.l.google.com` via DNS-over-UDP.
2. STUN lookup against each resolved address.
3. Resolve `dns.google` via DNS-over-UDP.
4. DNS-over-UDP lookup for `www.example.com` against each
   resolved `dns.google` address.
5. DNS-over-HTTPS lookup for `www.example.com` against each
   resolved `dns.google` address.

This is intentionally a hardcoded sequence. The goal is to
start collecting data while the tool is still being developed.

## Contextual tags

After the STUN lookup, the scan injects the discovered reflexive
addresses (`reflexiveAddrV4`, `reflexiveAddrV6`) as contextual
tags via `netstack.ContextWithTags`. All subsequent measurements
carry these tags in every structured log line, so each span is
self-describing: an analyst looking at a Parquet extract can see
which public IP was in use at measurement time without needing
to correlate with a separate STUN span.

Tags flow through `context.Context`, which avoids data races
when the measurer is shared across goroutines. The alternative
of storing tags as mutable state on `SondaMeasurer` was rejected
because `Resolver`, `HTTPTransport`, and other types all hold a
pointer to the same measurer.

## Error handling

By default, the scan is lenient: individual measurement
failures are logged as warnings and the scan continues. All
spool entries are written regardless of success or failure.
The exit code is 1 if any step failed, 0 otherwise.

The `--fail` flag makes the scan exit immediately on first
failure, for use in CI or smoke tests.

## What this is not

This is not the final measurement orchestration layer. The
sequence, the endpoints, and the error handling will change.
This is a bootstrap: a way to put the existing building
blocks to use and start generating real data while we figure
out what the orchestration layer should look like.

## Known limitations

- The DNS resolver used to look up `dns.google` defaults to
  `8.8.8.8:53`, which means we resolve Google DNS using Google
  DNS. A future version will parse `resolv.conf` or accept a
  system resolver override.

- IPv6 measurements fail on hosts without IPv6 connectivity.
  The lenient error handling means this is not fatal, but the
  spool will contain failed entries for all IPv6 targets.

- No scheduling or concurrency. The scan runs sequentially
  and exits. Timing lives in systemd, not in sonda.
