# Design: `sonda scan`

## Purpose

Periodic scan of specific network endpoints. Intended to be
triggered by a systemd timer (or equivalent). Runs a sequence
of steps and stores all results in the spool.

## Step-based architecture

The scan is structured as an ordered list of steps, each
described by a `singleStep` struct with three fields:

- `name` — human-readable label for logging.
- `run` — selects the runner (e.g., `stun`, `dns-over-udp`,
  `dns-over-https`, `https`, `extract`, `load`, `gc`).
- `with` — operation-specific parameters passed as a
  string-to-string map.

This is modeled after GitHub Actions: a step declares *what*
to run and *with* which parameters, but the ordering is
explicit and imperative. The design starts imperative (the
user spells out exactly what happens and in what order) with
the option to add semantic compression later.

Each `run` value maps to a `stepRunner` implementation
through a registry built at startup. The registry is
a `map[string]stepRunner` constructed in `Main`, where each
runner receives its dependencies explicitly. Runners that
perform network measurements (stun, dns-over-udp,
dns-over-https, https) receive a logger, measurer, resolver,
and shared state. Runners that invoke subprocesses (extract,
load, gc) receive a logger, the testable environment, and
the relevant directory paths.

## Config file

Steps can be loaded from a YAML config file via the
`--config-file` flag. When omitted, a minimal built-in
default is used (STUN lookup, one DNS check, garbage
collection). The full workflow ships as
`/etc/sonda/scan/default.yml`, declared as a Debian
`conffile` so that local edits are preserved on upgrade.

The systemd unit passes
`--config-file /etc/sonda/scan/default.yml`. A sysadmin
can edit that file to add or remove measurement targets without
recompiling the binary. The config file path is a regular
flag, not a hardcoded path, so the scan command works
identically in development without any config file.

The YAML structure mirrors the Go types:

```yaml
steps:
  - name: "STUN lookup"
    run: stun
    with:
      server: stun.l.google.com
```

## Shared state

Steps communicate through a mutex-protected
`map[string]string` called `sharedState`. The STUN runner
writes reflexive addresses into this map using `SetTag`:

- `reflexiveAddrV4` — the IPv4 reflexive address.
- `reflexiveAddrV6` — the IPv6 reflexive address.

Subsequent measurement runners read the current tags and
inject them into the context via `netstack.ContextWithTags`.
This means every measurement span carries the public IP
that was in use at measurement time, without needing to
correlate with a separate STUN span.

Using a map with `SetTag` (which overwrites by key) avoids
a footgun with the earlier slice-based design: running STUN
multiple times would balloon the tag list, and a `ClearTags`
primitive would nuke tags set by other steps.

## Contextual tags

Tags flow through `context.Context`, which avoids data races
when the measurer is shared across goroutines. The alternative
of storing tags as mutable state on `SondaMeasurer` was
rejected because `Resolver`, `HTTPTransport`, and other types
all hold a pointer to the same measurer.

## Error handling

By default, the scan is lenient: individual step failures
are logged as warnings and the scan continues. The `--fail`
flag makes the scan exit immediately on first failure, for
use in CI or smoke tests.

A bad or missing config file is fatal: the scan logs an
error and exits with code 2 (usage error convention),
since there are no steps to execute.

Unknown `run` values in the step list are logged as warnings
and skipped, rather than being fatal. This allows a config
file written for a newer version to degrade gracefully on
an older binary.

## Measurement runners

Each measurement runner resolves the target hostname to
addresses, then performs the measurement against each
resolved address individually. This means:

- DNS runners (dns-over-udp, dns-over-https) resolve the
  server hostname, then query each address.

- The HTTPS runner resolves the host, then issues a GET
  against each resolved address. This measures all IPs
  rather than stopping at the first success.

- The STUN runner resolves the server, performs lookups
  against each address, and writes the discovered reflexive
  addresses into shared state.

## Housekeeping runners

Three runners invoke `sonda` subcommands as subprocesses:

- `extract` — runs `sonda spool extract` to produce
  Parquet extracts from the spool. The `max_age` parameter
  controls how far back to look (default: `1h`).

- `load` — runs `sonda metrics load` to load extracts
  into daily metric files.

- `gc` — runs `sonda spool gc` to remove old spool
  entries. The `max_age` parameter controls the retention
  window (default: `6h`).

These are subprocesses rather than library calls because
the extract and gc commands have their own flag parsing,
error handling, and logging. Keeping them as subprocesses
means the scan does not need to duplicate that logic,
at least as long as this code is a prototype.

Directory paths (`spool-dir`, `metrics-dir`) are provided
to these runners at construction time from the scan
command's own flags, not from the step's `with` map. The
`with` map carries only step-specific parameters like
`max_age`.

## Known limitations

- The DNS resolver used to look up measurement targets
  defaults to `8.8.8.8:53`, which means we resolve Google
  DNS using Google DNS. A future version will parse
  `resolv.conf` or accept a system resolver override.

- IPv6 measurements fail on hosts without IPv6 connectivity.
  The lenient error handling means this is not fatal, but the
  spool will contain failed entries for all IPv6 targets.

- No scheduling or concurrency. The scan runs sequentially
  and exits. Timing lives in systemd, not in sonda.
