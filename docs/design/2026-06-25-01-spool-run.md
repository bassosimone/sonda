# Design: `sonda spool run`

## Purpose

Bernstein-style command wrapper. Executes an arbitrary command and
collects its stdout, stderr, and exit code into a structured spool
directory. Does not inspect the command or its output.

## Spool layout

- Sharded by UUIDv7 timestamp prefix: `<spool-dir>/XXXX/X/X/<spanID>/`.
The first four characters change every ~50 days, the next two every
~3 days and ~4.7 hours respectively, so leaf directories hold a
manageable number of entries.

- Each entry contains: `argv.json` (command that was executed),
`stdout.txt`, `stderr.txt`, `exitcode.txt`.

## Atomicity

- All work happens in a `<spanID>.tmp` directory. When the command
has finished and all files are closed, the directory is atomically
renamed to the final `<spanID>` path.

- Consumers only see complete entries. Orphaned `.tmp` directories
are crash artifacts — safe to garbage-collect.

## Span ID propagation

- The wrapper generates a UUIDv7 and passes it to the child process
via the `SONDA_SPAN_ID` environment variable.

- Native sonda measurement tools honor it (flag > env > random).
Foreign tools ignore it harmlessly — the spool directory path
still provides the correlation.

## Graceful shutdown

- On timeout, sends SIGINT first. Escalates to SIGKILL after a
5-second grace period via `cmd.Cancel` / `cmd.WaitDelay`.

- No process group detachment: CTRL+C propagates naturally to the
child via the default process group.

## Separation of concerns

- The wrapper does not know what command it runs, does not parse
output, and does not retry or loop. Scheduling lives above it,
processing lives below it. Each layer is independently testable.

## Testability

- All OS interactions (file I/O, process execution, environment
access) go through `testable.Environ` function fields. The entire
flow can be tested without real processes or filesystem operations.
